SGLang #25820：DeepSeek-V4 NVFP4 MoE 接线图

识别 hybrid checkpoint -> 分流 FusedMoE -> FlashInfer TRTLLM routed 执行

GitHub PR #25820

先看全局：它不是一个单独 kernel，而是一条接线链路

DeepSeek-V4 NVFP4 checkpoint 是混合量化：普通层 FP8，MoE expert NVFP4。这个 PR 的核心是让框架识别它，并让 FusedMoE 走正确的 FP4 backend。

config.json -> nvfp4_moe_meta -> HybridFp8NvFp4Config -> FusedMoE NVFP4 -> flashinfer_trtllm_routed
记忆点：普通层不换路，只有 MoE expert 被分流到 NVFP4 快路径。

旧理解容易错在哪里

错误理解 1

“整个模型都是 NVFP4。”实际是 mixed precision，普通层仍复用 FP8 逻辑。

错误理解 2

“只要 loader 能读权重就行。”实际还要 quant method、runner backend、routing scale 都对齐。

错误理解 3

“性能来自一个神秘 kernel。”实际是让 MoE expert 进入已有 FP4 routed MoE 快路径。

代码入口地图

配置识别

1model_config.py 读取 mixed precision + NVFP4detect
2抽出 group_size 和 exclude_modulesmeta
3loader.py 构造 HybridFp8NvFp4Configwrap

执行分流

4modelopt_quant.py 判断 layer 是否 FusedMoEdispatch
5FusedMoE 走 ModelOptNvFp4FusedMoEMethodFP4 MoE
6deepseek_v4_hook.py 默认选择 routed runnerbackend

两个容易漏的模型语义

routing scale

topk_weights x routed_scaling_factor -> MoE combine

在 hash_topk.py 中补上，不然 expert 输出权重语义会错。

SwiGLU clamp

GEMM1 -> clamp -> SwiGLU -> GEMM2

在 FlashInfer TRTLLM FP4 MoE quant info 中传递 gemm1_clamp_limit。

最后背这张卡

目标DeepSeek-V4 mixed FP8+NVFP4 checkpoint 能被 SGLang 正确识别。
代码动作用 Hybrid config 让普通层走 FP8、FusedMoE 走 NVFP4。
其他模型要做补配置解析、权重映射、quant method、backend kernel、correctness 和 benchmark。
