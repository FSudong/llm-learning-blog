vLLM #44667 可视化：NVFP4 MoE 权重边解边乘

packed byte -> nibble -> scale -> tl.dot，不整包解压

GitHub PR #44667

先记住这一个矛盾

MoE 很稀疏：当前 token 只命中少数 experts。但旧路径会先把所有 expert 的 NVFP4 权重都解压成 BF16 大 tensor，然后再做 GEMM。

E0未命中，也被解压
E1命中，需要计算
E2未命中，也被解压
E3命中，需要计算
核心：新 PR 不再整包解压所有 expert，而是在 GEMM tile 内只解当前命中的 expert tile。

三个词

packed NVFP4

一个 uint8 byte 里塞两个 4-bit 权重值：low nibble 和 high nibble。

inline dequant

不生成完整 BF16 权重，只在 kernel tile 内临时解码。

tl.dot

解码和缩放后的 tile 立刻进入矩阵乘，不写回 HBM。

一步一步看：一个 byte 怎么变成 GEMM 的权重

旧路径 vs 新路径

旧路径：先整包解压

1读取所有 experts 的 packed NVFP4 权重全量
2dequantize_to_dtype 生成 BF16 权重HBM 写
3TritonExperts.apply 再读取 BF16 权重HBM 读
4只对命中的 experts 做 GEMM稀疏性来晚了

新路径：边解边乘

1moe_align_block_size 找到命中 expert block只命中
2load 当前 expert 的 packed weight tiletile
3拆 nibble、decode、scalekernel 内
4直接 tl.dot，不生成完整 BF16 权重少 HBM

结合代码看最小逻辑

旧：先完整反量化

w1_dequant = dequantize_to_dtype(
    tensor_fp4=w1,
    tensor_sf=w1_scale,
    global_scale=w1_global_scale,
    dtype=hidden_states.dtype,
)

super().apply(
    hidden_states=hidden_states,
    w1=w1_dequant,
    w2=w2_dequant,
)

新：kernel 内边解边 dot

raw_bytes = tl.load(b_ptrs)

low_nibble = raw_bytes & 0x0F
high_nibble = (raw_bytes >> 4) & 0x0F

low = _e2m1_inline(low_nibble)
high = _e2m1_inline(high_nibble)

scale = block_scale * global_scale
b = tl.trans(tl.interleave(low * scale, high * scale))
accumulator = tl.dot(a, b, acc=accumulator)

为什么小 batch / decode 特别赚

旧路径的固定成本

即使只有 1 个 token，旧路径也可能先把整层 MoE 的 packed 权重解压出大块 BF16 中间 tensor。

E0 E1 E2 E3 -> BF16 大 tensor

新路径的按需成本

当前 batch 命中哪些 expert，就只解这些 expert 当前 GEMM tile 需要的 byte。

E1 tile E3 tile -> tl.dot

PR 给出的效果

下面是 PR 描述里的 Kimi-K2.6-NVFP4 TP=8 MoE forward microbench。注意这是 MoE forward/kernel 级收益，不等于整服务端端到端 10x。

tokens unfused fused speedup
1 1.843 ms 0.171 ms 10.80x
2 1.850 ms 0.176 ms 10.51x
16 2.053 ms 0.274 ms 7.48x

最后只背这张卡

旧路径

FP4 权重先全部解压成 BF16 大 tensor，再让 MoE GEMM 读取。

新路径

只读取命中 expert 的 packed tile，在 kernel 内拆 nibble、decode、scale。

为什么快

不把完整 dequantized weight 写入 HBM，少全量工作，稀疏性更早生效。
