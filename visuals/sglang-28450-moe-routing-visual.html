核心：旧代码先生成中间表再改表；新 PR 在一个 Triton kernel 里直接写最终表。

三个词

shared expert

每个 token 都会额外走的公共专家，像给 top-k 表再加一列。

DeepEP remap

把 logical expert id 翻译成多 GPU dispatch 后端需要的 physical id。

padding mask

固定 shape 里有假 token，假 token 的 expert weight 要变成 0。

一步一步看数据怎么变

旧路径 vs 新路径

旧路径：表被改好几次

1append shared expert1 launch
2arange 造 shared indexsmall
3floor-div: e // localsmall
4add: e + gapsmall
5fill/copy 写回 top-k 表small

新路径：一个 kernel 写最终答案

1load topk ids/weightssame launch
2在寄存器里算 remapALU
3直接写 routed physical idssame launch
4直接写 shared physical idssame launch
5padding rows 用单 Triton kernelfused

arange 到底在干什么

`arange` 就是生成一排编号。它本身不复杂，但在 GPU eager 模式下会变成一个独立 kernel；后面的比较和写入也常常是独立 kernel。

代码形状

indices = torch.arange(0, 8, device="cuda")
mask = indices >= num_token_non_padded
topk_weights[mask, :] = 0.0

数据形状

indices
0123 4567
mask
FFFF TTTT
为什么手写 Triton：PyTorch eager 看到的是三四个通用张量操作；PR 作者知道业务语义是“第 4 行之后全部填 0”，所以能写成一个专用 kernel。

DeepEP remap 的具体画法

假设有 4 个普通 routed experts，2 张 GPU，每张 GPU 有 2 个 routed experts 和 1 个 shared expert。DeepEP physical layout 会在每个 rank 的 routed block 后面留 shared slot。

Rank 0 physical ids

P0=E0 P1=E1 P2=S0

Rank 1 physical ids

P3=E2 P4=E3 P5=S1
num_local_routed = 2 E0 -> 0 + 0//2 = P0 E1 -> 1 + 1//2 = P1 E2 -> 2 + 2//2 = P3 E3 -> 3 + 3//2 = P4

为什么从 device memory 读取真实 token 数

不适合 graph capture

GPU scalar -> CPU .item() -> 同步等待

CPU 每步都读 GPU 上的 `num_token_non_padded`，会破坏高性能 serving 想要的固定 replay 流程。

适合 graph capture

固定 kernel -> tl.load(scalar) -> 内部判断

kernel 每次 replay 都一样，只是 GPU memory 里的 scalar 值变了。形状固定，行为动态。

结合代码看最小逻辑

旧 padding mask

indices = torch.arange(0, rows, device=x.device)
x[indices >= num_token_non_padded, :] = 0.0

新 padding mask

row = tl.program_id(0)
n_valid = tl.load(num_token_non_padded_ptr)
if row >= n_valid:
    x[row, :] = fill_value

旧 append + remap

topk = append_shared(topk)
topk = remap_for_deepep(topk)

新 fused kernel

ids = load(topk_ids[row])
ids = ids + ids // num_local_routed
store(out_ids[row, :K], ids)
store(out_ids[row, K:], shared_id_base + s)

最后只背这张卡

MoE 做什么

router 给所有专家打分，但真正 MLP 只跑 top-k，再额外跑 shared expert。

PR 改什么

把 append shared expert 和 DeepEP remap 从多步改表，变成一次写最终表。

为什么快

少了很多小 GPU kernel launch，decode 阶段每层都少做固定开销。
