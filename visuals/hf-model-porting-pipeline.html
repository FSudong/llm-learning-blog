第一张图：HF 模型包不是可直接执行的高性能 serving 模型

config.json tokenizer safetensors chat template -> 框架识别 -> 高性能 runtime
核心：权重只是原材料。框架还要知道 forward 怎么走、权重怎么映射、KV cache 怎么管、并行怎么切。

第二张图：先判断要不要写代码

1打开 config.json，看 architectures入口
2architecture 已在框架 registry 里？直接跑
3结构相同但名字不同？注册 alias
4结构相似但有小差异？复制最近模型改
5新 attention / MoE / VLM / quant？实现新模型和测试

第三张图：vLLM 链路

architectures -> ModelRegistry -> models/my_model.py -> forward + compute_logits + sample -> load_weights
内置上游放进 vllm/model_executor/models，改 registry.py。
外部实验用 plugin 调 ModelRegistry.register_model。
最常见坑QKV/gate_up fused 权重映射、prefix、TP/PP。

第四张图：SGLang 链路

architectures -> 扫描 srt/models -> EntryClass -> RadixAttention + ForwardBatch -> LogitsProcessor
注册方式模型文件末尾写 EntryClass = MyModel。
vLLM port 差异Attention 换 RadixAttention，forward 带 forward_batch。
测试先 HF reference，再 sglang.bench_one_batch --correct。

第五张图：TensorRT-LLM 链路

HF weights -> convert -> TRT-LLM checkpoint -> trtllm-build -> TensorRT engine -> runner / serve
TensorRT-LLM 的新模型接入更像“编译一个 engine”：除了模型定义，还要处理 checkpoint 转换和 build profile。

第七张图：上线前 checklist

configarchitecture、heads、RoPE、MoE、quant 字段全部记录。

weightsQKV、gate/up、tie embedding、scale 名字映射正确。

最后只背这句话

接入模型的主线永远是：config.json -> architectures -> registry -> model file -> load_weights -> correctness。MoE、量化、VLM、TensorRT engine 都是在这条链路上继续加复杂度。
