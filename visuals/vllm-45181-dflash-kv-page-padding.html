vLLM #45181：DFlash 混合 KV page size

target/draft page 不能整除时，用 physical padding + as_strided 对齐

GitHub PR #45181

先看问题：target 和 draft 的 KV page 不一样大

DFlash speculative decoding 里，target model 和 draft model 可以有不同 KV head size。比如 target=192，draft=128，同样 block_size 下 page size 是 3:2，不是整数倍。

target KV head 192 -> page 192 单位 vs draft KV head 128 -> page 128 单位
192 / 128 = 1.5，不能靠把 block_size 乘一个整数来对齐。

旧方案只会“整除时放大 block_size”

能整除：可以

小 page 512 x2 大 page 1024

把小模型 block_size 乘 2，page size 就能一致。

不能整除：旧代码失败

draft 128 x1.5? target 192

block_size 不能乘 1.5，所以旧逻辑只能报错。

新方案：逻辑 block 不变，物理 page 补空洞

draft 有效数据 128 padding 64 = 物理 page 192 
if max_page_size % layer_page_size == 0:
    new_spec = replace(layer_spec, block_size=block_size * ratio)
elif isinstance(layer_spec, AttentionSpec) \
     and layer_spec.indexes_kv_by_block_stride:
    new_spec = replace(layer_spec, page_size_padded=max_page_size)
else:
    raise NotImplementedError

padding 之后，kernel 怎么跳过空洞？

关键是不能简单 .view()。PR 用 torch.as_strided 只改“下一页从哪里开始”的 stride。

物理内存

block0 data pad block1 data pad

padding 真实存在于物理 page 尾部。

逻辑 view

block0 stride=page_size_padded block1

view 看到的是有效 KV，跳页时跨过 padding。

page_stride = kv_cache_spec.page_size_bytes // dtype_size
strides = list(torch.empty(permuted_kv_cache_shape).stride())
strides[num_blocks_dim] = page_stride

kv_cache = torch.as_strided(
    kv_raw_tensor.view(dtype),
    size=permuted_kv_cache_shape,
    stride=tuple(strides),
)

为什么要 backend 声明支持

AttentionBackend -> indexes_kv_by_block_stride() -> 允许 padding
如果 kernel 不按 block stride 找下一页，它可能会把 padding 当成 KV 数据读进去，所以必须显式 gate。

测试覆盖的布局

测试点为什么重要
target 192 / draft 128DFlash 典型 3:2 非整数 page ratio

HND stride order验证不同维度排列下 stride 仍正确

最后背这张卡

目标DFlash target/draft KV page size 不一致也能统一管理。
关键不能整除时设置 page_size_padded，不改逻辑 block_size。
安全条件backend 必须支持按 block stride 访问 KV cache。
