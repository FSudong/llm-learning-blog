vLLM #45424：async H2D 之前先 pin memory

non_blocking=True 不是魔法，CPU 源内存要满足异步传输条件

GitHub PR #45424

先看核心矛盾

看起来异步

torch.from_numpy(x).to(
    device,
    non_blocking=True,
)

但如果 CPU tensor 不是 pinned memory，底层可能仍需要同步或 staging。

更接近真异步

torch.from_numpy(x)
    .pin_memory()
    .to(device, non_blocking=True)

CPU 源内存 page-locked 后，H2D copy 更容易和 GPU compute overlap。

普通 CPU memory vs pinned CPU memory

普通 CPU memory

CPU tensor -> 可能 staging / sync -> GPU

操作系统可能移动或换页，GPU DMA 传输条件不稳定。

pinned memory

pinned CPU tensor -> async H2D -> GPU

page-locked 内存更适合非阻塞上传。

PR 的两类代码动作

局部显式 pin

gpu_buffer.copy_(
    torch.from_numpy(req_id_per_token)
        .pin_memory(),
    non_blocking=True,
)

适合已有 GPU buffer，只需要把 CPU source 变成 pinned。

统一 helper

indices = async_tensor_h2d(
    indices,
    device=self.device,
)

把 CPU tensor 创建、pin、dtype、device、H2D copy 收敛到一个工具。

一个容易忽略的同步点：GPU 输出长度推断

pooling 路径里，PR 用 CPU 已知的 prompt_lens_cpu.sum() 显式给出 output_size，避免 CPU 再去等 GPU 推断长度。

# old: CPU 构造 segment_ids 再上传
segment_ids = torch.repeat_interleave(
    torch.arange(num_seqs, dtype=torch.long),
    prompt_lens_cpu,
).to(device, non_blocking=True)
# new: GPU 上 repeat，但输出长度 CPU 已知
prompt_lens = async_tensor_h2d(
    prompt_lens_cpu,
    device=device,
    dtype=torch.int64,
)
segment_ids = torch.repeat_interleave(
    torch.arange(num_seqs, device=device),
    prompt_lens,
    output_size=int(prompt_lens_cpu.sum()),
)
这和避免 .sum().item() 是同一类思想：热路径里不要让 CPU 等 GPU 回答小数字。

影响范围为什么很广

attention metadataquery_start_loc、seq_lens、doc ids 等小 tensor。
spec decodetoken indices、ngram proposer GPU buffer。
DeepSeek sparse MLAreq_id_per_token 这类 metadata buffer。
structured outputbitmask / index tensor 上传。

PR 没给 speedup 数字，那为什么重要？

1vLLM decode 热路径有很多小 metadata H2D频繁
2如果每次隐式同步，会削弱 graph/overlap 收益sync
3统一 pinned H2D，让“async”路径更可信cleanup

最后背这张卡

目标避免可能的 GPU/CPU stream sync。
核心CPU -> GPU non_blocking copy 前，source tensor 要 pinned。
迁移搜 .to(device, non_blocking=True) 和 copy_(..., non_blocking=True)。
